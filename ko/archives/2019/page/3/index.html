JSP

JSP(Java Server Pages)는 HTML에 Java 코드를 삽입하여 동적으로 웹 페이지를 생성하는 언어다. 아래는 JSP의 예제다.

JSP의 동작 원리

JSP는 다음과 같은 과정으로 처리된다.

  1. Web container는 JSP 파일을 Servlet으로 변환한다.
    • MyPage.jsp > MyPage_jsp.java
  2. Servlet 내부에서 JSP 파일의 내용을 출력한다. MyPage_jsp.java의 코드 일부를 살펴보자.

요약하자면 JSP는 MyPage.js > MyPage_jsp.java > MyPage_jsp.class > HTML 순서대로 처리된다.

JSP tag

JSP tag는 JSP 문서 안에 Java 코드를 삽입할 때 사용한다.

1. 자바 코드

<% %>를 사용하여 자바 코드를 추가할 수 있다.

2. 주석

주석은 <%-- --> 형태로 사용한다.

<%! %>는 변수를 선언하거나 함수를 정의할 때 사용한다.

4. 표현식

JSP에서 무언가를 출력하려면 out.println()을 사용한다. <%! %>는 변수를 선언하거나 함수를 정의할 때 사용한다.

(1-3) taglib

태그 라이브러리를 사용할 때 사용한다.

[Servlet, JSP] Servlet 이란?

2019/01/08 카테고리 Spring, Servlet, JSP

Servlet

Servlet은 Java를 사용하여 웹 페이지를 동적으로 생성하는 서버측 컴포넌트다. 모든 Servlet은 HttpServlet클래스를 상속한다.

Servlet URL Mapping

Servlet은 두 가지 방법으로 URL Mapping을 할 수 있다.

방법 1. @WebServlet 어노테이션

방법 2. web.xml

Servlet의 동작 순서


  1. Client로 부터 요청이 들어오면 WAS는 Web Container와 Context를 생성한다.
  2. Web Container는 요청에 대한 HttpServletRequest, HttpServletResponse를 생성한다.
  3. Web Container는 web.xml의 Servlet 매핑 정보를 확인하여 어떤 Servlet을 생성할지 결정한다.
  4. Web Container는 Servlet이 한 번도 실행된 적이 없으면 Servlet의 인스턴스를 생성한다. 실행된 적이 있으면 Servlet을 재사용한다.
  5. Web Container는 요청을 처리하기 위한 Thread를 생성한다.
  6. Servlet의 doGet(), doPost(), service()가 호출되며 인자로 HttpServletRequest, HttpServletResponse 객체가 전달된다.
  7. 로직이 수행되면 HttpServletRequest, HttpServletResponse 객체를 소멸시키고 Thread를 종료한다.

Servlet의 생명주기

Web Container는 Client의 요청에 따라 Servlet을 생성, 관리, 소멸한다. 이때 Servlet 클래스에 정의된 생명주기 함수가 호출된다.


  • Constructor: Servlet 인스턴스가 생성될 때 최초 한 번 호출되는 생성자
  • init(): Servlet 인스턴스가 생성될 때 최초 한 번 호출되는 생명주기 함수
  • doGet(), doPost(), service(): Client에서 요청할 때 마다 호출되는 생명주기 함수
    • doGet(): HTTP GET 방식으로 요청할 때 호출
    • doPost(): HTTP POST 방식으로 요청할 때 호출
    • service(): doGet()과 doPost()가 정의되지 않았을 때 호출
  • destroy(): Servlet 인스턴스가 소멸될 때 한번 호출

Servlet 선처리, 후처리

@PostConstruct, @PreDestroy 어노테이션을 임의의 메서드에 추가하여 선처리, 후처리를 할 수 있다.

doGet()

HTTP GET 방식으로 요청을 하면 doGet()메소드가 호출된다.

doPost()

HTTP POST 방식으로 요청하면 doPost()메소드가 호출된다.

왜 빌드 변형이 필요할까?

안드로이드에서는 하나의 프로젝트를 여러 버전의 앱으로 빌드할 수 있습니다. 그렇다면 왜 여러 버전의 앱으로 빌드를 해야할까요?

(1) 스마트폰 앱 vs. 태블릿 앱

스마트폰용 앱과 태블릿용 앱을 출시한다고 가정합시다. 두 앱은 기능은 동일하고 레이아웃만 약간 다릅니다. 이러한 경우 두 개의 프로젝트를 생성하여 각각 관리한다면 매우 비효율적일 것입니다.

(2) 무료 앱 vs. 유료 앱

이번엔 무료 앱과 유료 앱을 출시한다고 가정합시다. 두 앱은 레이아웃은 동일하며, 기능만 약간 다릅니다. 이러한 경우에도 두 개의 프로젝트를 생성하는 것은 비효율적입니다.

(3) 개발용 앱 vs 출시용 앱

마지막으로 개발용 앱과 출시용 앱이 있다고 가정합시다. 프로가드(ProGuard)는 앱 용량을 축소하고 코드를 난독화합니다. 다만 프로가드를 적용하면 빌드 시간이 길어지는데 굳이 디버그용 앱에 적용할 필요가 있을까요? 이러한 경우 출시용 앱에만 적용되도록 할 수 있습니다.

빌드 변형

하나의 프로젝트가 여러 버전으로 빌드되면 각각의 앱을 빌드 변형(Build Variant)이라고 합니다. 빌드 변형을 만드는 방법에는 두 가지가 있습니다.

  • 빌드 타입(Build Type)
  • 프로덕트 플래이버(Product Flavor)

빌드 타입

안드로이드 스튜디오 프로젝트를 생성하면 debug, release라는 두 개의 빌드 타입(BuildType)이 자동으로 생성됩니다.

각 빌드 타입에 대한 설정은 모듈 수준의 build.gradle의 buildTypes 섹션에서 할 수 있습니다. 프로젝트를 처음 생성하면 아마 아래와 같은 모습일 것입니다.

[Servlet, JSP] Server, Web Server, WAS 차이점

2019/01/04 카테고리 Spring, Servlet, JSP

Server

Server는 Client에 정보나 서비스를 제공하는 컴퓨터 시스템이다. 서버에는 보통 Linux, Unix, Windows 운영체제가 탑재된다.

Web Server

  • Web Server는 웹 브라우저 같은 클라이언트로부터 HTTP 요청을 받아들이고 HTML 문서와 같은 웹 페이지를 반환하는 프로그램이다.
  • Web Server의 대표적인 제품군에는 Apache, Nginx, IIS가 있다.
  • Web Server는 파일 시스템에 저장된 파일(HTML, CSS, JavaScript, 이미지, 파일 등)을 단순히 클라이언트에게 전달한다. 이를 정적 컨텐츠를 제공한다고 한다.

WAS(Web Application Server)

WAS는 데이터베이스, 파일 시스템, 네트워크 등에서 데이터를 읽어와 다양한 로직을 수행한 후 적절한 HTML 파일을 생성하여 클라이언트에게 전달한다. 이를 동적 컨텐츠를 제공한다고 한다.

WAS의 대표적인 제품군에는 Apache Tomcat, Oracle WebLogic, IBM Websphere, Tmax JEUS, Red Hat Jboss가 있다. 제품군에 따라 다르지만 WAS는 크게 세 가지 요소로 구성된다.

  • Web Container
    1. JSP파일을 Servlet으로 변환한다.
    2. Servlet의 생성, 소멸 등의 생명주기를 관리한다.
    3. HttpServletRequest, HttpServletResponse의 생성, 소멸 등의 생명주기를 관리한다.
    4. Web Container는 Servlet Container라고도 불린다.
  • Web Server: Web Container가 동적으로 생성한 웹 페이지를 Client에게 제공한다.
  • EJB Container:
    1. EJB(Enterprise Java Bean)의 생명주기를 관리한다.
    2. EJB는 한국에서는 Spring Framework에 밀려서 잘 사용되지 않는다.

Oracle WebLogic, Red Hat Jboss 같은 제품군은 세 가지 요소를 모두 포함한다. 그러나 Apache Tomcat은 Web Server와 Web Container만을 포함한다.
