리눅스 권한 관리

리눅스 시스템에서 사용자는 파일과 디렉토리를 소유할 수 있다. ls명령어에 -l옵션을 추가하면 파일이나 디렉토리의 접근 권한을 확인할 수 있다.

출력 결과의 의미는 다음과 같다.

  1. 파일의 종류
  2. 권한
  3. 심볼릭 링크 수
  4. 해당 파일을 소유한 사용자
  5. 해당 파일을 소유한 그룹. 별도로 지정하지 않으면 파일 소유자가 속한 그룹
  6. 용량
  7. 생성일
  8. 파일 이름

앞의 10글자를 좀 더 자세히 살펴보자.

  1. 파일의 종류
  2. 파일 소유자의 권한
  3. 파일 소유자가 속한 그룹의 권한
  4. 그 외 사용자의 권한

파일의 종류

속성 파일 종류
- 일반 파일
d 디렉토리
l 심볼릭 링크
c 문자 특수 파일
b 블록 특수 파일
s 소켓
p 파이프

권한 속성

파일과 디렉토리에는 r(read), w(write), x(Execution) 값으로 권한을 부여할 수 있다.

속성 파일 디렉토리
r 파일을 열거나 읽을 수 있다. ls 명령어로 디렉토리의 내용물을 나열할 수 있다.
w 파일을 쓰거나 수정할 수 있다. 디렉토리 내의 파일들을 생성, 삭제, 이름 변경할 수 있다.
x 파일을 실행할 수 있다. cd 명령어로 디렉토리에 들어갈 수 있다.

권한 변경

chmod명령어를 사용하면 권한을 변경할 수 있다. 이 명령어는 루트 사용자나 파일 소유자만 실행할 수 있다.

(1) 8진법 표현

8진법 750은 2진법 111101000에 해당하며, 이는 rwxr-x---와 동일하다.

(2) 기호 표현

기호 표현은 다음과 같이 사용한다. 아래 예제는 파일 소유자에게 실행 권한을 부여한다는 의미다.

이제 사용자와 관련된 기호에 대해 살펴보자.

기호 의미
u user의 약자, 파일 소유자
g group의 약자, 그룹 소유자
o other의 약자, 기타 사용자
a all의 약자, u, g, o를 모두 포함

퍼미션 추가, 삭제, 설정과 관련된 기호는 다음과 같다.

기호 의미
+ 뒤에 나열된 권한을 추가
- 뒤에 나열된 권한을 제거
= 뒤에 나열된 권한만 적용하고 나머지는 제거

몇 개의 예제를 첨부한다.

계정 종류

리눅스에는 크게 세 가지 계정이 존재한다.

  • 루트 계정: 모든 권한을 가지고 있는 최고 관리자. 0번 UID를 부여받는다.
  • 시스템 계정: 시스템 운영상 필요한 계정. 일반적으로 로그인이 불가능하며, 1~100 사이의 UID를 부여받는다.
  • 일반 계정: 레드햇 계열은 100에서 499 사이, 데비안 계열의 경우 100에서 999 사이의 UID를 부여받는다.

계정 목록 확인

모든 계정은 /etc/passwd파일에 정의되어있다.

계정은 다음과 같은 형태로 정의된다.

  1. 계정 이름
  2. 패스워드: 값이 x인 경우 패스워드는 /etc/shadow에 암호화되어 저장된다.
  3. 사용자 ID: 모든 계정은 유일한 사용자 ID(UID)를 갖는다.
  4. 그룹 ID: 모든 계정은 적어도 하나의 그룹에 포함된다.
  5. 설명: 계정에 대한 설명
  6. 홈 디렉토리: 사용자의 홈 디렉토리
  7. 기본 쉘: 로그인 시 적용되는 기본 쉘

계정 비밀번호

계정의 비밀번호는 암호화되어 etc/shadow에 저장된다.

그룹

사용자는 적어도 하나의 그룹(Group)에 포함된다. 그룹에 대한 정보는 /etc/group파일에 정의된다.

그룹은 다음과 같은 형태로 정의된다.

  1. 그룹 이름
  2. 그룹 패스워드: 새로운 계정을 그룹에 가입시킬 때 사용할 수 있다. 선택사항이다.
  3. 그룹 ID: 모든 그룹은 유일한 ID(GID)를 가진다.
  4. 그룹 멤버: 그룹에 포함된 모든 유저 목록. 콤마로 유저를 구분한다.

계정 추가

리눅스 설치 시 기본적으로 루트 계정과 시스템 계정만 존재하고 일반 계정은 존재하지 않는다. useradd명령어와 다양한 옵션을 사용하여 계정을 추가할 수 있다.

user1, user2 계정을 추가해보자.

/etc/passwd파일에서 추가된 계정을 확인할 수 있다.

/home디렉토리에서 사용자의 홈 디렉토리가 생성된 것도 확인할 수 있다.

계정 비밀번호 설정

passwd명령어로 계정 비밀번호를 설정할 수 있다.

계정 속성 변경

usermod명령어를 사용하면 사용자의 홈 디렉토리, 그룹, 기본 쉘 등 여러가지 속성을 변경할 수 있다. user1의 그룹을 group1에서 group2로 변경해보자.

계정 삭제

userdel명령어를 사용하면 사용자를 삭제할 수 있다. 계정 뿐만 아니라 홈 디렉토리 등 관련된 모든 정보를 삭제하려면 옵션 -r을 추가한다.

현재 계정 확인

환경변수 USER의 값을 출력하여 현재 세션에 접속한 사용자를 확인할 수 있다.

whoami명령어를 사용할 수도 있다.

그룹 추가

groupadd명령어를 사용하면 그룹을 추가할 수 있다.

그룹 확인

groups명령어로 현재 사용자가 속한 그룹을 확인할 수 있다.

그룹 삭제

groupdel 명령어로 그룹을 삭제할 수 있다.

su 명령어

Switch User의 약자인 su명령어를 사용하면 현재 계정을 로그아웃을 하지 않고 다른 계정으로 전환할 수 있다.

루트 계정에서 일반 계정으로 전환할 때는 비밀번호를 입력할 필요가 없다.

일반 계정에서 다른 일반 계정으로 전환할 때는 비밀번호를 입력한다.

계정 이름을 입력하지 않으면 루트 계정으로 전환한다.

-을 사용하면 계정을 전환하면서 환경변수까지 적용한다. 아래 예제는 환경변수가 적용되었기 때문에 현재 경로가 계정의 홈 디렉토리로 이동했다.

logout명령어를 사용하면 전환 전 계정으로 돌아갈 수 있다.

exit 명령어를 사용해도 된다.

sudo 명령어

Superuser do의 약자인 sudo명령어를 사용하면 다른 계정의 권한을 빌려 명령어를 실행할 수 있다. 계정이 전환되진 않으며, 보통 일반 계정이 관리자 계정의 권한을 잠시 빌려서 명령어를 실행할 때 사용한다.

예제를 살펴보자. 일반 계정에서 apt-get update를 실행하면 권한 오류가 발생한다.

관리자는 일반 계정에게 sudo명령어를 사용할 수 있는 권한을 부여해야한다. 관리자 계정에서 /etc/sudoers파일을 다음과 같이 수정하면 된다.

이제 일반 계정에서 sudo명령어를 사용할 수 있다. 이때 관리자 계정이 아닌 일반 계정의 비밀번호를 입력한다.

id 명령어

id명령어를 사용하면 사용자의 ID를 확인할 수 있다.

  • uid: 사용자의 ID (User ID)
  • gid: 사용자가 포함된 주 그룹의 ID (Group ID)
  • groups: 사용자는 주 그룹 외에도 다른 그룹에 속할 수 있다.

[Linux] 리눅스 파일 시스템 구조

2020/08/04 카테고리 Infra, DevOps, Linux

거의 대부분의 리눅스 배포판은 리눅스 파일시스템 계층 표준인 Linux Filesystem Hierarchy Standard을 준수한다. 따라서 다음 디렉토리 구조를 가진다.

여기서 중요한 몇 가지 디렉토리를 살펴보자.

이름 설명
/ 최상위 디렉토리인 루트 디렉토리
/boot/ 부팅에 필요한 파일들이 존재한다.
/bin/ 운영체제가 제공하는 명령어 바이너리. 모든 사용자들이 사용할 수 있다. ls, cat, cp, rm, ps 같은 명령어가 이 곳에 존재한다.
/sbin/ 운영체제가 제공하는 명령어 바이너리. 시스템 관리자만 사용할 수 있다. useradd, userdel, usermod 같은 명령어가 이 곳에 존재한다.
/lib /bin/과 /sbin의 명령어 바이너리가 필요로하는 라이브러리 파일
/root 루트 사용자의 홈 디렉토리
/home 일반 사용자의 홈 디렉토리
/etc 시스템 전반의 환경설정 파일이 존재한다. 계정 설정과 관련된 /etc/passwd, /etc/shadow, /etc/group이 이 곳에 존재한다.
/var 상대적으로 자주 변하는 파일들이 존재한다.
/tmp 다양한 프로그램에서 만들어지는 임시 파일들이 저장된다. 시스템이 재부팅될 때 마다 이 디렉토리가 비워진다.
/usr Universal Resource의 약자. 일반 사용자가 사용하는 모든 프로그램과 자원 파일들이 존재한다.
/opt Optional의 약자. 일반 사용자가 사용하는 프로그램이 설치된다.
/dev 리눅스는 장치를 파일로 관리한다. 연결된 장치가 이 곳에서 파일 형태로 존재한다.
/mnt 디스크처럼 저장 기능이 있는 장치는 파일시스템에 마운트할 수 있다. 이 곳에는 마운트 포인트가 존재한다.
/media USB 드라이버, CD-ROM 같은 휴대용 장치가 연결될 때의 마운트 포인트가 이 곳에 존재한다.

다음 디렉토리는 내부를 좀 더 살펴볼 필요가 있다.

/var

/var에는 로그 데이터, 사용자 메일 같은 가변 데이터 파일이 저장된다. 대표적으로 var/log에는 시스템이 기록하는 로그 파일이 위치한다.

/usr

/bin에는 운영체제가 제공하는 명령어 바이너리가 위치한다. 반면 /usr에는 사용자가 설치한 명령어 바이너리가 위치한다.

이름 설명
/usr/bin 사용자가 설치한 앱이 위치한다. 모든 사용자가 사용할 수 있다.
/usr/sbin 사용자가 설치한 앱이 위치한다. 관리자만 사용할 수 있다.
/usr/lib /usr/bin 또는 /usr/sbin에 설치한 앱이 의존하는 라이브러리가 존재한다.
/usr/local/bin /usr/bin과 유사하지만 Package Manager가 아닌 사용자가 직접 컴파일한 앱이 위치한다. 모든 사용자가 사용할 수 있다.
/usr/local/sbin /usr/bin과 유사하지만 Package Manager가 아닌 사용자가 직접 컴파일한 앱이 위치한다. 관리자만 사용할 수 있다.
/usr/local/lib /usr/local/bin 또는 /usr/local/sbin에 설치한 앱이 의존하는 라이브러리가 존재한다.

/opt

/opt는 사용자가 설치한 명령어 바이너리가 위치한다는 점에서 /usr/bin과 유사하다.

보통 UNIX 계열에서는 사용자가 설치한 애플리케이션이 /opt에 위치한다. 반면 대부분의 Linux 배포판에서 설치한 애플리케이션은 /usr/bin에 위치힌다.

[Android] 의존성 주입(DI) - Hilt 사용법

2020/08/03 카테고리 Android, DI, Hilt

안드로이드 의존성 주입 라이브러리들

안드로이드에서 사용하는 대표적인 의존성 주입 라이브러리는 세 가지 입니다.

  • Koin
  • Dagger 2
  • Hilt

Koin은 코틀린 환경에서 사용할 수 있는 의존성 주입 라이브러리 입니다. 다른 의존성 주입 라이브러리에 비해 상대적으로 배우고 쉽습니다. 그러나 Koin은 런타임에 리플렉션을 통해 의존성을 주입해주므로 성능이 저하될 수 있습니다.

Dagger나 Hilt는 의존성 주입에 어노테이션을 사용합니다. 어노테이션은 컴파일 타임에 코드로 변환되므로 컴파일 타임은 길어질 수 있으나 런타임에 성능이 저하되지 않습니다. 따라서 큰 규모의 프로젝트에서는 Dagger나 Hilt가 권장됩니다.

설정

먼저 hilt-android-gradle-plugin 플러그인을 프로젝트 수준의 build.gradle 파일에 추가합니다.

그 다음 모듈 수준의 build.gradle 파일 상단에 hilt-android-gradle-plugin 플러그인을 적용합니다.

마지막으로 다음 종속성을 추가합니다.

Hilt는 자바 8를 사용하므로 자바 8 사용을 설정합니다.

사용 방법

(1) @HiltAndroidApp

Hilt를 사용하려면 Application를 정의하고 @HiltAndroidApp 어노테이션을 추가해야합니다.

@HiltAndroidApp을 추가하면 Singleton Component가 생성됩니다. Component는 스프링의 IoC Container와 유사하며, 주입할 객체를 관리하고 필요한 곳에 주입하는 역할을 합니다. Singleton Component는 앱 전체의 생명주기와 함께하며 단일 객체로 유지되는 Component입니다. Singleton Component에 설치한 의존성은 앱 어디에서든 주입받을 수 있게됩니다. 참고로 Component는 의존성을 주입해준다는 점에서 Injector라고도 불립니다.

(2) @AndroidEntryPoint

Hilt는 여러 의존성을 관리하고 적절하게 주입해줍니다. Hilt로 부터 의존성 주입을 받으려면 @AndroidEntryPoint 어노테이션을 추가해야합니다.

@AndroidEntryPoint는 다음과 같은 안드로이드 클래스에 추가할 수 있습니다.

  • Application
  • Activity
  • Fragment
  • View
  • Service
  • BroadcaseReceiver

예제를 살펴보겠습니다. YourActivity에 @AndroidEntryPoint를 추가합시다.

액티비티에 @AndroidEntryPoint를 추가하면 ActivityComponent가 생성됩니다.

참고로 생성된 Component들은 아래와 같은 계층구조를 갖습니다.

따라서 SingletonComponent의 하위에 ActivityComponent가 그래프의 형태로 생성됩니다. 또한 SingletonComponent에 설치한 모듈에 포함된 의존성은 ActivityComponent에서도 사용할 수 있습니다. (즉, 부모 Component에 설치한 의존성은 자식 Component에서 사용할 수 있습니다.)

(3) 의존성 주입

이제 ThemeManager 객체를 YourActivity에 주입하려고 합니다.

의존성을 주입하려면 주입할 객체를 Hilt에 등록해야합니다. 이를 위해 ThemeManager의 생성자 앞에 @Inject 어노테이션을 추가하면 됩니다. 이 객체는 Component에 설치됩니다.

ViewModel객체를 Hilt에 등록할 때는 @HiltViewModel과 @Inject를 사용합니다.

모듈

때로는 주입하려는 클래스의 생성자 앞에 @Inject를 붙일 수 없는 경우가 있습니다. 이러한 경우 모듈(Module)을 사용할 수 있습니다.

(1) 외부 라이브러리의 객체를 주입

외부 라이브러리에서 제공하는 클래스는 직접 코드를 수정할 수 없습니다. 따라서 필요한 생성자 앞에 @Inject를 붙일 수도 없습니다. 이러한 경우 모듈을 사용할 수 있습니다.

참고로 @Singleton어노테이션을 추가하면 객체를 싱글톤으로 유지할 수 있습니다.

(3) UseCase 객체 주입

Clean Architecture에서는 UseCase를 ViewModel에 주입합니다.

환경변수

쉘(Shell)은 쉘 세션이 유지되는 동안 다양한 변수를 유지한다. 이 변수를 환경변수라고 한다.

env명령어로 쉘이 관리하고 있는 모든 환경변수를 확인할 수 있다.

환경설정파일

리눅스 시스템에 로그인하면 일련의 환경설정 스크립트가 읽혀진다. 이 스크립트 파일에 여러가지 환경변수를 설정하게된다. 환경설정 스크립트는 쉘 세션의 종류에 따라 다르게 적용된다.

(1) 로그인 쉘 세션에서의 환경설정파일

로그인 쉘 세션에서는 다음과 같은 순서로 환경설정 스크립트가 적용된다.

파일 설명
/etc/profile 모든 사용자에게 적용되는 환경설정파일
~/.profile 특정 사용자에게 적용되는 환경설정파일

만약 사용자의 시작 쉘이 bash라면 다음과 같은 순서대로 적용된다.

파일 설명
/etc/profile 모든 사용자에게 적용되는 환경설정파일
~/.bash_profile 특정 사용자에게 적용되는 환경설정파일
~/.bash_login ~/.bash_profile이 없을 경우 적용
~/.profile ~/.bash_profile도 없고 ~/.bash_login도 없을 경우 적용

사용자의 시작 쉘이 bash라면 로그아웃할 때 ~/.bash_logout이 실행된다.

(2) 비로그인 쉘 세션에서의 환경설정파일

비로그인 쉘 세션에서의 환경설정파일을 살펴보자.

사용자의 시작 쉘이 bash인 경우 환경설정 스크립트가 다음과 같은 순서대로 적용된다.

파일 설명
/etc/bash.bashrc 모든 사용자에게 적용되는 환경설정파일
~/.bashrc 특정 사용자에게 적용되는 환경설정파일

(3) ~/.bashrc 파일

일반 사용자 관점에서 가장 중요한 파일은 ~/.bashrc파일이다. 로그인 쉘 세션이든 비로그인 쉘 세션이든 이 파일이 적용되기 때문이다.

비로그인 쉘 세션에서는 시작 쉘이 bash라면 기본적으로 이 파일이 적용된다. 로그인 쉘 세션에서는 보통 ~/.bash_profile, ~/.bash_login, ~/.profile에서 ~/.bashrc을 적용하도록 구현한다.

ViewModel 주입

Koin을 사용하면 MVVM아키텍처에서 ViewModel을 쉽게 주입할 수 있습니다. LoginViewModel을 LoginActivity에 주입해보겠습니다.

ViewModel을 위한 모듈을 정의합니다. 이 때 factory()나 single() 대신 viewModel()을 사용합니다.

액티비티에서는 by viewModel()을 통해 ViewModel을 주입받을 수 있습니다.

모듈을 Application클래스에 추가합니다.

뷰모델의 생성자에 파라미터가 있는 경우

Koin을 사용하지 않는 프로젝트에서는 생성자의 파라미터가 있는 뷰모델을 생성하기 위해 ViewModelFactory를 정의해야합니다. 그러나 Koin을 사용하면 ViewModelFactory를 자동으로 생성해주기 때문에 직접 ViewModelFactory를 정의할 필요가 없습니다.
